## 2024-05-27 - Remove Hardcoded Admin Credentials Fallback
**Vulnerability:** The application contained hardcoded default administrative passwords (`DEFAULT_ADMIN_PASSWORD` and `DEFAULT_PASSWORD_HASH`) in both the Python backend (`server/core/auth.py`) and JS backend (`api/[...path].js`). If environment variables were not explicitly provided, these fallbacks allowed unauthorized access using a known default credential.
**Learning:** Default fallbacks for critical security configuration variables undermine fail-safe defaults and create universal backdoors if the environment is misconfigured. Empty string edge cases during hash comparisons (in both Python's `hmac.compare_digest` and Node.js's `crypto.timingSafeEqual`) meant setting the fallback to an empty string without explicitly rejecting empty password inputs could also lead to authentication bypass.
**Prevention:** Avoid default fallbacks for secrets. Enforce that configuration values must be explicitly provided in the environment, and always validate input to reject empty credentials before performing cryptographic hash comparisons.

## 2024-05-27 - Fix SSRF and Event Loop Blocking in Custom Transport
**Vulnerability:** The proxy endpoint was vulnerable to Server-Side Request Forgery (SSRF) because it fetched arbitrary user-provided URLs without validating if they resolved to internal network addresses. When attempting to fix it using a custom `httpx.AsyncHTTPTransport` to check resolved IP addresses, using `socket.gethostbyname()` synchronously blocked the asyncio event loop. Furthermore, it failed to handle IPv6 (raising a `gaierror`), meaning an attacker could bypass the filter entirely by using an IPv6 address.
**Learning:** In asynchronous environments (like FastAPI), never use blocking network operations (e.g., `socket.gethostbyname`). Using them introduces Denial of Service (DoS) vulnerabilities by blocking the main thread. Additionally, security controls must fail securely (deny access) rather than failing open (allowing access) when exceptions like DNS resolution errors occur. Finally, ensure both IPv4 and IPv6 addresses are explicitly validated against blocked ranges.
**Prevention:** Always use non-blocking asynchronous DNS resolution (e.g., `await loop.getaddrinfo(...)`). Ensure the check logic handles both IPv4 and IPv6, and catch all resolution errors to explicitly deny the request rather than letting it fall through to a potentially vulnerable default state.
